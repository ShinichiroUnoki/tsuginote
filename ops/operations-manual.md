# TsugiNote 1人運用マニュアル

**作成日**: 2026年3月29日
**対象**: サービス運営者（1人体制）
**設計思想**: 日次15分、週次1時間、月次2時間で回せる運用

---

## 1. 日次タスク（15分以内）

毎朝実施。慣れれば5-10分で完了。

### チェックリスト

```markdown
## 日次チェック — YYYY/MM/DD

### ヘルスチェック（3分）
- [ ] サービス稼働確認: [ステータスページ] を確認 → 🟢正常 / 🔴異常
- [ ] エラー通知（Sentry → Slack）: 未対応エラーの有無 → 0件 / _件
- [ ] レスポンスタイム: P95が200ms以下か → 正常 / 要調査

### KPI確認（5分）
- [ ] 昨日の新規サインアップ: _件
- [ ] WAW（直近7日）: _
- [ ] AI生成回数（昨日）: _回
- [ ] 解約通知: 0件 / _件（あれば理由確認）

### カスタマーサポート（5分）
- [ ] 問い合わせ確認: 0件 / _件
- [ ] 未回答の問い合わせに返信（24時間以内ルール）

### 異常時のみ
- [ ] エラーが発生していたら → 「3. 障害対応ランブック」を参照
- [ ] 解約があったら → 解約理由を記録、パターン分析
```

### 自動化済みの項目（手動確認不要）

| 項目 | 自動化方法 | 通知先 |
|------|----------|--------|
| サーバーヘルスチェック | ヘルスチェックエンドポイント（/health）の定期監視 | Slack |
| エラー検知 | Sentry → Slack通知 | Slack #errors |
| DB自動バックアップ | 日次バックアップジョブ（06:00 JST） | ログのみ |
| 依存パッケージ更新 | Dependabot / Renovate | GitHub PR |
| SSL証明書更新 | Let's Encrypt 自動更新 | 失敗時のみSlack |

## 4. 障害対応ランブック

### 4-1. 障害レベル定義

| レベル | 定義 | 対応時間 | 例 |
|--------|------|---------|------|
| **P0: Critical** | サービス全停止、データ損失リスク | 即時対応（15分以内着手） | DB接続不能、全API 5xx |
| **P1: High** | 主要機能の障害 | 1時間以内着手 | AI生成不能、認証エラー |
| **P2: Medium** | 一部機能の障害 | 24時間以内着手 | 検索精度低下、UIバグ |
| **P3: Low** | 軽微な問題 | 次回開発サイクルで対応 | 表示崩れ、非クリティカルなエラー |

### 4-2. 主要障害シナリオと対応手順

#### シナリオ1: サービス全停止（P0）

```
検知: Slack #alerts に「Health check failed」通知

手順:
1. ステータスページを「障害発生中」に更新
2. ホスティングプロバイダーのダッシュボードを確認
   → サーバー停止していたら再起動
   → プロバイダー側障害ならステータスページで確認
3. アプリケーションログを確認
   → エラーの特定
   → 直近のデプロイが原因ならロールバック: [手順は下記]
4. DB接続確認
   → 接続できない場合、DBプロバイダーのステータス確認
5. 復旧確認後、ステータスページを「復旧済み」に更新
6. 障害報告をまとめる（原因・影響・対策）
```

#### シナリオ2: AI生成が動かない（P1）

```
検知: Sentry に AI API呼び出しのエラーが多発 / ユーザー報告

手順:
1. LLM APIプロバイダーのステータス確認
   - OpenAI: https://status.openai.com/
   - Anthropic: https://status.anthropic.com/
   → プロバイダー障害の場合、フォールバックプロバイダーに切り替え
2. APIキーの有効性確認（期限切れ、クォータ超過）
3. ネットワーク接続確認
4. Ollama利用時: Ollamaプロセスの生存確認、GPUメモリ確認
5. ユーザーへの通知:「AI生成機能に一時的な障害が発生しています」
```

#### シナリオ3: DB障害（P0）

```
検知: アプリログに「connection refused」/ Sentry大量エラー

手順:
1. DBプロバイダーのステータス確認
2. 接続数の上限確認（コネクションプール枯渇の可能性）
   → 上限到達時はアプリケーション再起動
3. ディスク容量確認
4. 復旧しない場合:
   a. 最新バックアップからのリストア手順を準備
   b. リストア実行（テスト環境で先に確認）
5. データ整合性チェック
```

#### シナリオ4: Stripe決済障害（P1）

```
検知: Stripe Webhook失敗通知 / ユーザーから「課金できない」報告

手順:
1. Stripe Dashboard でイベントログ確認
2. Webhookエンドポイントの到達性確認
3. Stripe側障害: https://status.stripe.com/ 確認
   → Stripe側の場合は復旧を待つ（通常1-2時間以内）
4. Webhook失敗の場合、手動で再送トリガー
5. ユーザーへの影響確認（二重課金が起きていないか）
```

#### シナリオ5: デプロイ後の不具合（P1-P2）

```
検知: デプロイ後にエラー率上昇 / ユーザー報告

手順:
1. 直近のデプロイ内容を確認（git log -5）
2. エラー内容を特定（Sentryでスタックトレース確認）
3. 原因が直近デプロイと断定できたら:
   a. ロールバック実行
   b. ロールバック後の動作確認
4. 原因を修正して再デプロイ
5. デプロイ後の監視強化（30分間エラー率を注視）
```

### 4-3. ロールバック手順

```bash
# GitHub Actionsでデプロイしている場合
# 1. 直前の正常なコミットを特定
git log --oneline -10

# 2. そのコミットに戻すPRを作成 or 直接revert
git revert HEAD
git push origin main
# → GitHub Actions が自動デプロイ

# 3. 動作確認
# ヘルスチェックエンドポイントを叩いて確認
curl https://api.tsuginote.com/health
```

## 6. セキュリティインシデント対応

### 6-1. インシデントレベル定義

| レベル | 定義 | 対応 |
|--------|------|------|
| **S0: Critical** | データ漏洩、不正アクセス確認 | 即時対応。必要に応じてサービス停止 |
| **S1: High** | 脆弱性発見（悪用の可能性あり） | 24時間以内にパッチ適用 |
| **S2: Medium** | 脆弱性発見（悪用リスク低） | 1週間以内に対応 |
| **S3: Low** | セキュリティ改善提案 | 次回開発サイクルで検討 |

### 6-2. データ漏洩発生時の対応手順

```
【即時対応（発覚から1時間以内）】

1. 影響範囲の特定
   - どのデータが漏洩したか
   - 何件のユーザーが影響を受けるか
   - いつから発生していたか

2. 封じ込め
   - 漏洩経路の特定と遮断
   - 必要に応じてサービスの一時停止
   - 関連するAPIキー・認証情報のローテーション

3. 証拠保全
   - アクセスログの保存（改ざん防止のためコピーを取得）
   - 関連するシステムの状態をスナップショット

【24時間以内】

4. 影響を受けたユーザーへの通知
   - 何が起きたか
   - どのデータが影響を受けたか
   - ユーザーが取るべきアクション（パスワード変更等）
   - 当社の対応状況

5. 個人情報保護委員会への報告
   ※個人情報保護法に基づき、個人データの漏洩等が発生した場合:
   - 速報: 発覚から3-5日以内
   - 確報: 発覚から30日以内（不正アクセスの場合は60日以内）
   - 報告先: 個人情報保護委員会
   - 報告方法: オンライン報告フォーム

【1週間以内】

6. 根本原因分析
   - なぜ発生したか（技術的原因）
   - なぜ防げなかったか（プロセス的原因）

7. 再発防止策の策定・実施

8. インシデントレポートの作成
```

### 6-3. 不正アクセス検知時の対応

```
【検知パターン】
- 同一IPからの大量ログイン試行（ブルートフォース）
- 通常と異なるIPアドレスからの管理者アクセス
- 異常なAPI呼び出しパターン

【対応手順】
1. 該当IPアドレスをブロック
2. 影響を受けたアカウントのセッションを無効化
3. アカウント所有者に通知（パスワード変更を推奨）
4. アクセスログの詳細分析
5. WAFルールの追加（必要に応じて）
```

### 6-4. 脆弱性発見時の対応

```
【外部からの報告（Responsible Disclosure）】
1. 報告者への受領確認（24時間以内）
2. 脆弱性の検証
3. パッチの開発・テスト
4. パッチ適用
5. 報告者への対応完了通知
6. 必要に応じてユーザーへの通知

【Dependabotによる検知】
1. Critical/High: 24時間以内にPRマージ
2. Medium: 1週間以内にPRマージ
3. Low: 次回メンテナンスウィンドウで対応
```
